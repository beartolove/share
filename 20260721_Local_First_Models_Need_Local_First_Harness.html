重點摘要

  • 這篇文挑戰「原廠 harness 最懂自家模型」的直覺：同一顆 Kimi K3、同一套 Terminal-Bench 2.1，Maka 自報分數高於官方 Kimi Code。
  • 差距不是神祕技巧，而是 harness 工程：刪除過期 tool result、精簡 prompt 與工具面、用 benchmark / trace / CI 紀律持續迭代。
  • local-first 模型不能只看權重與量化；真正能做事的是「模型 × harness」配對，harness 本身會消耗 token、注意力、時間與資安信任。
  • 地端場景尤其需要做減法：更薄的 prompt、更少工具、更好的 context prune、timeout 策略與本機 session 控制。
  • Maka 數據仍屬自報、尚待第三方復現；但它已提醒：開源、可控、輕量化 harness 可能比多買一張顯卡更划算。

提醒：Maka 分數為團隊自報，目前尚無第三方復現；分享時建議把它視為「值得追蹤的工程訊號」，而不是已定論的 benchmark 事實。

Harness 做減法的四個工程重點

做法具體內容為什麼重要
Context prune把過期 tool result 從 context 砍掉減少雜訊，讓模型不要在舊結果中迷路
薄 prompt / 少工具避免 20KB 產品 prompt 與全量工具面地端模型 attention 與速度都更吃緊，必須做減法
Benchmark-driven CI跑分 → 看 trace → 改進 → 重跑 → 分數不掉才合併把 harness 當成可測、可回歸、可治理的工程產品
Local-first trustsession / workspace / 設定盡量留在本機避免把 local model 的私密 codebase 又交回雲端 harness

為什麼 local-first 時代更重要

地端模型的瓶頸不只在模型權重、量化與 GPU，而在整個 agent stack。雲端 frontier model 能承受厚 prompt、全量工具與高 token 消耗；地端模型則會把每個多餘工具、每 KB prompt、每段過期 context 都放大成速度、成本與成功率問題。

因此，local-first 的真正選型不是「哪一顆模型最強」，而是「模型 × harness」能不能在本機限制下穩定完成任務，並保住資料邊界。
